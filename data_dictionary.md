# 📖 Data Dictionary & Workflow (ฉบับภาษาไทย)
**โครงการ:** ระบบสารบรรณและลงนามเอกสารอิเล็กทรอนิกส์ (edoc67)

เอกสารนี้อธิบายถึงความหมายของตาราง ข้อมูลในคอลัมน์ และขั้นตอนการทำงาน (Workflow) ของระบบ

---

## 🗂️ 1. พจนานุกรมข้อมูล (Data Dictionary)

### 1.1 ตาราง `documents` (ข้อมูลเอกสารหลัก)
ใช้สำหรับเก็บหัวข้อและรายละเอียดเบื้องต้นของเอกสารที่นำเข้าสู่ระบบ

| คอลัมน์ | ความหมาย | หมายเหตุ |
| :--- | :--- | :--- |
| `doc_id` | รหัสเอกสาร (PK) | |
| `inst_id` | รหัสสถาบัน | รองรับ Multi-Tenant |
| `doc_no` | เลขที่หนังสือ | เช่น "ศธ 0605/..." |
| `doc_name` | ชื่อเรื่องเอกสาร | หัวข้อของหนังสือ |
| `doc_from` | ผู้ส่ง/หน่วยงานต้นทาง | |
| `doc_type_id` | ประเภทเอกสาร | เชื่อมกับตาราง `document_types` |
| `doc_uploader`| ผู้ที่นำเอกสารเข้าระบบ | |

### 1.2 ตาราง `user_signatures` (คลังลายเซ็นต์)
เก็บลายเซ็นต์ของผู้ใช้งาน (หนึ่งคนมีได้หลายแบบ)

| คอลัมน์ | ความหมาย | หมายเหตุ |
| :--- | :--- | :--- |
| `sig_id` | รหัสลายเซ็นต์ (PK) | |
| `user_id` | รหัสผู้ใช้งาน (FK) | เชื่อมกับ user.user_id |
| `sign_path` | ที่อยู่ไฟล์ลายเซ็นต์ | Path รูปภาพ |
| `is_primary` | ลายเซ็นต์หลักหรือไม่ | 1=ใช่, 0=ไม่ใช่ |
| `created_at` | วันที่สร้าง | |

### 1.3 ตาราง `sign_doc` (การควบคุม Workflow)
ตารางนี้สำคัญที่สุด ใช้ควบคุมสถานะการไหลของเอกสาร

| คอลัมน์ | ความหมาย | หมายเหตุ |
| :--- | :--- | :--- |
| `sign_doc_id` | รหัสลำดับการลงนาม | |
| `doc_id` | รหัสเอกสาร | |
| `user_id` | ผู้เริ่มเรื่อง (งานสารบรรณ) | |
| `dep_id` | ฝ่ายที่รับผิดชอบ (รอง ผอ.) | ระบุฝ่ายที่จะให้ รอง ผอ. พิจารณา |
| `doc_status` | **สถานะรวมของเอกสาร** | `pending`, `approve`, `reject` |
| `sign_sarabun` | สถานะของงานสารบรรณ | `pending`, `approve` |
| `sign_codirector`| สถานะของ รอง ผอ. | `pending`, `approve`, `reject` |
| `sign_director` | สถานะของ ผอ. | `pending`, `approve` |

### 1.3 ตาราง `sign_detail` (รายละเอียดการวางตำแหน่ง)
เก็บข้อมูลตำแหน่งร่องร่อยการลงนาม หรือตราปั้มบนไฟล์ PDF

| คอลัมน์ | ความหมาย | หมายเหตุ |
| :--- | :--- | :--- |
| `sign_doc_id` | เชื่อมกับ Workflow | |
| `page_num` | เลขหน้าของ PDF | เริ่มต้นที่ 1 |
| `x_pos`, `y_pos` | พิกัดตำแหน่งบนกระดาษ | หน่วยเป็น Pixel |
| `sign_txt` | ข้อความที่เขียน/ปั้ม | เช่น "ทราบ", "พิจารณาดำเนินการ" |
| `sign_pic` | รูปภาพลายเซ็นต์ | Path ของไฟล์รูป |
| `sign_by` | ผู้ที่ทำการลงนาม/ปั้ม | |

## 🔄 2. ขั้นตอนการทำงาน (Workflow Stages)

ระบบแบ่งการทำงานออกเป็น 3 ขั้นตอนหลัก:

### **ขั้นตอนที่ 1: งานสารบรรณ (Sarabun)**
1. เจ้าหน้าที่อัปโหลดไฟล์ PDF
2. เลือกฟอร์มเกษียณ และเขียนข้อความนำเสนอ (เช่น "เรียน ผอ. เพื่อโปรดพิจารณา...")
3. **กำหนดฝ่าย (Co-Director):** เลือกฝ่ายที่จะส่งเรื่องให้ รอง ผอ. ตรวจสอบ
4. ระบบจะบันทึก `sign_sarabun = 'approve'` และส่งเรื่องต่อ

### **ขั้นตอนที่ 2: รองผู้อำนวยการ (Co-Director)**
1. รอง ผอ. แต่ละฝ่ายจะเห็นเอกสารในเมนู "จัดการเอกสาร (รองฯ)"
2. ตรวจสอบรายละเอียดและลงนาม/ปั้มตราประทับ
3. บันทึก `sign_codirector = 'approve'` เพื่อส่งให้ ผอ. หรือ `reject` หากต้องการตีกลับ

### **ขั้นตอนที่ 3: ผู้อำนวยการ (Director)**
1. ผอ. จะเห็นเอกสารที่ผ่านการกลั่นกรองจาก รองฯ แล้วเท่านั้น
2. ทำการลงนามขั้นสุดท้าย
3. เมื่อ ผอ. ลงนาม สถานะ `sign_director` จะเป็น `'approve'` และ `doc_status` จะถือว่า **เสร็จสมบูรณ์**

## 🚦 3. สรุปสถานะต่างๆ (Status Glossary)

| สถานะ | ความหมาย | 📍 ใช้ในคอลัมน์ |
| :--- | :--- | :--- |
| **`pending`** | อยู่ในระหว่างรอการดำเนินการ | ทุกคอลัมน์สถานะ |
| **`approve`** | ผ่านการตรวจสอบ/ลงนามแล้ว | ทุกคอลัมน์สถานะ |
| **`reject`** | ไม่ผ่าน/ถูกตีกลับ | `doc_status`, `sign_codirector` |
| **`finish`** | เอกสารสถานะสุดท้าย (ลงนามครบถ้วน) | `doc_status` |

---
> [!NOTE]
> ในกรณีของ **Multi-Department** ระบบจะสร้างแถวใน `sign_doc` แยกตาม `dep_id` ทำให้ รอง ผอ. แต่ละฝ่ายเห็นเอกสารใบเดียวกันและดำเนินการแยกกันได้โดยไม่รบกวนกันครับ
